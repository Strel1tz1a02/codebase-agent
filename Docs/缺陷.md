现在我有完整的图景了。以下是系统性的分析：

---

## 项目不足之处

按严重程度排序，从功能断裂到可优化点。

---

### 一、功能断裂（Critical）

**1. Graph 工具调用路径完全不工作**√

这是当前版本最严重的功能问题。在 `runtime/runs.py:_run_graph` 中：

```python
state = create_initial_state(...)
state["rag_index"] = index
if self.chat_model is not None:
    state["chat_model"] = self.chat_model
result = self.graph.invoke(state)
```

`AgentGraphState` 中声明了 `tools` 和 `tool_executor` 字段，但 `_run_graph` 从未将它们注入 state。后果：

- **`plan_tool_use`** — `state.get("tools", [])` 永远是空列表，LLM 无论如何都无法感知到工具的存在
- **`execute_tools`** — `state.get("tool_executor")` 永远是 `None`，每次工具调用都返回 `"No tool executor configured."`
- **`plan_tool_use` 的 prompt** 中硬编码了 4 个工具名称，但这些工具从未真正绑定到 model 调用

结果：RAG 检索 → 回答这条路径勉强能走，但整个工具调用子图（read_file、search_code、retrieve_code、repo_summary）形同虚设。

**2. RAG 索引每次都完整重建，无持久化**

`src/rag/indexing.py` 的 `build_project_index()`：

```python
file_records = load_code_files(repo_path)
chunks = chunk_code_files(file_records)
documents = [build_document(...) for chunk in chunks]
vector_store = build_local_vector_store(build_embeddings(config.embeddings))
_index_documents(vector_store, documents)
```

每次调用都从头扫描文件、切块、构建 Document、重新 embedding、写入向量存储。对于中等规模仓库（>500 个文件），每次 `index_project()` 都可能耗时数十秒。没有磁盘缓存，服务重启后必须重新索引。

---

### 二、架构设计缺陷（High）

**3. Session 多轮对话上下文丢失**√

`_run_graph` 只将当前问题注入 state：

```python
state = create_initial_state(..., question=run.question)
```

之前在同一个 session 中的历史对话（前几轮的 question/answer）从未拼接到 `state["messages"]` 中。这意味着每轮提问都是独立的，LLM 不知道之前讨论过什么。项目中存在 `_build_run_messages` （在后续版本中）但没有在 `_run_graph` 中被调用。

**4. Graph 同步阻塞 API**

`RuntimeService.ask()` 同步执行 `self.graph.invoke(state)`。当请求通过 FastAPI `POST /runs` 进入时，HTTP 线程会被阻塞直到 LangGraph 完成执行。对于复杂问题，请求可能等待 10-30 秒，导致：

- FastAPI 线程池耗尽
- HTTP 客户端超时
- 无法支持长时间运行的分析任务

**5. 无数据持久化**

`RuntimeStore` 是纯内存存储：

```python
@dataclass
class RuntimeStore:
    projects: dict[str, Project] = field(default_factory=dict)
```

服务重启后，所有 project / session / run / event 全部丢失。既然已有 HTTP API 对外暴露，这是实际服务化的关键缺失。

**6. API 错误处理策略不统一**√

- `routes_projects.py` 捕获自定义 `ProjectNotFoundError` 转 404
- `routes_sessions.py` 捕获 `KeyError` 转 404
- `routes_runs.py` 同捕获 `KeyError`，但通过 `app.state.not_found_from_key_error` 转换

同一类"资源不存在"错误有三种不同的处理模式。

---

### 三、流程健壮性不足（Medium）

**7. "invalid" 路由导致无限循环**√

`_normalize_next_step()` 在无法识别 LLM 输出时返回 `"invalid"`。`route_after_plan` 将其映射到 `"plan_next_step"`，这会再次调用 LLM，大概率又输出 "invalid"，形成**死循环**。没有计数器、没有兜底、没有 fallback。

**8. 校验逻辑形同虚设**

```python
def validate_answer(state):
    valid = bool(str(state.get("answer", "")).strip())
    reason = "" if valid else "empty answer"
```

任何非空字符串都算"校验通过"。没有对幻觉、漏引用、空话的回答做任何检查。

**9. `plan_tool_use` 硬编码工具列表**√

```python
"鍙敤宸ュ叿锛歳epo_summary銆乺ead_file銆乻earch_code銆乺etrieve_code銆俓n"
```

工具列表写在 prompt 字符串中，而非从 `src/tools/registry.py` 动态生成。如果新增或删除工具，必须同步修改 prompt。

**10. 未使用 LangChain 原生 Tool Calling**

`plan_tool_use` 把工具列表写在纯文本 prompt 中，让 LLM 回 JSON 文本然后 `json.loads()` 解析：

```python
tool_calls = _parse_tool_calls(
    _extract_model_content(chat_model.invoke(prompt))
)
```

没有使用 `chat_model.bind_tools(tools)` 的**原生 function calling**。这意味着：

- 不支持模型的工具 Schema 验证
- 输出格式不稳定（JSON 解析易失败）
- 不支持并行工具调用（部分模型原生支持）

---

### 四、工程化缺失（Low - Medium）

**11. 缺少项目管理能力**

- API：无 `GET /projects` 列出所有项目，无 `DELETE /projects/{id}`
- CLI：无 `list` / `projects` 命令
- CLI 的 `ask --project` 接受的是十六进制 `project_id`（如 `a1b2c3d4...`），而非用户友好的项目名称

**12. 配置无法通过 API 热更新**

`ModelConfig`、`EmbeddingConfig`、`RetrievalConfig` 只能在代码中构造，没有 API 端点查看或修改。如果需要更换模型或调整 top_k，必须重启服务。

**13. RuntimeService 配置传递断裂**√

`RuntimeService.index_project()` 调用 `build_project_index(project_id, repo_path)` **未传递 `self.config`**，导致 embedding 模型和向量存储总是使用默认配置，忽略用户自定义。

**14. `build_project_index` 参数未传递 config**√

`src/runtime/runs.py` 中：
```python
index = build_project_index(project.project_id, project.repo_path)
```
而 `build_project_index` 签名中有 `config: AppConfig | None = None`，但调用者没传。

**15. `finish` 节点状态覆盖**√

```python
if not str(state.get("status", "")):
    update["status"] = "completed"
```

当 `validate_answer` 设置了 `status="failed"` 时，`state.get("status", "")` 返回 `"failed"`（truthy），不覆盖。但真正的风险是：如果 `validate_answer` 从未运行（例如 graph 提前终止），`status` 可能是 `"running"`，此时 `not "running"` 为 False，也不覆盖。实际上问题在于 `"running"` 作为初始值（truthy）阻止了 finish 节点修正状态。

---

### 五、测试覆盖缺口

| 模块 | 是否有测试 | 风险 |
|---|---|---|
| `src/runtime/store.py` | ❌ | 无 |
| `src/runtime/sessions.py` | ❌ | 无 |
| `src/runtime/events.py` | ❌ | 无 |
| `src/core/paths.py` | ❌ | 路径逃逸的测试覆盖 |
| `src/core/errors.py` | ❌ | 无 |
| `src/utils/` | ❌ | 无 |
| 完整 Graph 端到端集成测试 | ❌ | 只有节点/路由的单元测试 |

`test_tools_path_safety.py` 和 `test_tools_langchain_registry.py` 依赖实际文件系统，使用临时目录隔离会更好。

---

### 六、代码层面问题

**16. 遗留代码未清理**

- `src/utils/file_io.py` — 从旧版本遗留，当前代码库中可能没有任何引用
- `src/utils/ignore.py` — 被 `tools/codebase.py` 使用，但部分逻辑与新的路径扫描冗余

**17. builder 中 condition_map 容错性问题**√

`builder.py` 中 `route_after_plan` 的 `condition_map` 包含 `"plan_next_step": "plan_next_step"` 的循环路由，但它没有兜底的路由目标指向 `synthesize_answer`。当 LLM 持续输出无效值时的唯一结果是无限循环。

---

### 总结

```
功能断裂（无法正常工作）：
  1. 工具调用路径未注入 state → 工具子图形同虚设
  2. RAG 索引无磁盘缓存 → 每次完整重建

架构缺陷：
  3. 多轮对话上下文丢失
  4. Graph 同步阻塞 API
  5. 无数据持久化
  6. API 错误处理不统一

健壮性不足：
  7. 无效路由导致无限循环
  8. 校验逻辑过于简单
  9. prompt 硬编码工具列表
  10. 未使用原生 Tool Calling

工程化缺失：
  11. 缺乏项目管理能力
  12. 配置不可热更新
  13. 配置传递断裂
  14-17. 其他代码质量问题
```

其中**第 1 点（工具路径断裂）** 是功能层面的最大问题——代码已经写了 4 个可用的工具和完整的工具注册表，但它们永远无法在 graph 中执行。
