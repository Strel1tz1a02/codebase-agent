## V1.5：LLM 项目问答实验版（不使用 RAG）

### 1. 当前模块目标

V1.5 的目标是在不引入 RAG、向量数据库和复杂 Agent 编排的前提下，先验证 LLM 对代码仓库的理解效果。

它要解决的问题：

- 复用 V1 的项目结构扫描结果。
- 读取少量关键文件内容。
- 将“项目结构 + 关键文件 + 用户问题”拼接成 prompt。
- 调用 LLM 生成基于当前仓库上下文的回答。

它在 `codebase-agent` 项目中的位置：

- 位于 V1 项目结构扫描器之后。
- 位于 V2 RAG 代码知识库之前。
- 是一个过渡版本，用来验证“LLM 直接看有限上下文能不能回答项目问题”。

---

### 2. 为什么先不使用 RAG

V2 原计划是：

```text
读取代码 -> 切分代码 -> embedding -> 向量索引 -> 检索 -> LLM 回答
```

但在早期阶段，可以先做一个更小的版本：

```text
扫描项目 -> 选择关键文件 -> 拼接 prompt -> LLM 回答
```

这样做的原因：

- 实现更简单，适合先验证效果。
- 不需要引入 FAISS、Chroma、embedding 等额外依赖。
- 可以先观察 LLM 对目录树、README、入口文件的理解能力。
- 后续如果发现上下文不够，再升级到 RAG。

---

### 3. 范围定义

`In Scope`（V1.5 要做）：

- 支持用户输入本地项目路径和自然语言问题。
- 复用 V1 的扫描结果，例如目录树、入口文件候选、主要目录、文件类型统计。
- 自动读取少量关键文件，例如：
  - `README.md`
  - `pyproject.toml`
  - `requirements.txt`
  - `package.json`
  - `src/main.py`
  - `main.py`
  - `app.py`
- 生成项目问答 prompt。
- 调用 LLM 返回回答。
- 回答中尽量附带使用到的文件路径。

`Out of Scope`（V1.5 不做）：

- 不做代码切分。
- 不做 embedding。
- 不做向量数据库。
- 不做语义检索。
- 不做多轮 Agent 状态管理。
- 不做 LangGraph 编排。
- 不让 LLM 自动调用工具。

---

### 4. 输入与输出契约

输入：

- `repo_path: str`：本地项目路径。
- `question: str`：用户自然语言问题。

中间上下文：

- `repo_summary: dict`：V1 扫描结果。
- `selected_files: list[str]`：被选中的关键文件路径。
- `file_contents: dict[str, str]`：关键文件内容。
- `prompt: str`：拼接后的 LLM 输入。

输出：

- `answer: str`：LLM 生成的回答。
- `used_files: list[str]`：本次回答提供给 LLM 的文件路径。

---

### 5. 推荐文件位置

建议新增或扩展以下文件：

```text
src/llm/client.py
src/prompts/templates.py
src/tools/file_tools.py
src/main.py
```

文件关系：

- `src/tools/file_tools.py`：复用 V1 的扫描能力，并补充读取关键文件的能力。
- `src/prompts/templates.py`：负责生成项目问答 prompt。
- `src/llm/client.py`：封装 LLM API 调用。
- `src/main.py`：提供 CLI 入口，支持 `--repo` 和 `--ask` 参数。

---

### 6. 推荐命令形式

示例：

```bash
python src/main.py --repo ./test_project --ask "这个项目的入口在哪里？"
```

执行流程：

```text
1. 扫描项目结构
2. 识别入口文件候选
3. 选择 README / 配置文件 / 入口文件等关键文件
4. 读取关键文件内容
5. 生成 prompt
6. 调用 LLM
7. 输出回答和参考文件路径
```

---

### 7. 函数级任务拆分

建议函数：

1. `select_context_files(scan_result: dict, repo_path: str) -> list[str]`

- 输入：V1 扫描结果、项目根路径。
- 输出：需要提供给 LLM 的关键文件路径列表。
- 作用：从项目中选择最值得读取的文件。
- 为什么需要：LLM 上下文有限，不能把整个仓库全部塞进去，需要先做一个简单的规则筛选。

2. `read_context_files(file_paths: list[str], max_chars_per_file: int) -> dict[str, str]`

- 输入：文件路径列表、每个文件最大读取字符数。
- 输出：文件路径到文件内容的映射。
- 作用：读取关键文件内容，并限制单文件长度。
- 为什么需要：避免超长文件撑爆 prompt。

3. `build_project_qa_prompt(repo_summary: dict, file_contents: dict[str, str], question: str) -> str`

- 输入：项目扫描摘要、关键文件内容、用户问题。
- 输出：完整 prompt 字符串。
- 作用：把结构化上下文转成 LLM 可以理解的输入。
- 为什么需要：prompt 质量直接影响回答质量，需要集中管理模板。

4. `ask_llm(prompt: str) -> str`

- 输入：prompt。
- 输出：LLM 回答。
- 作用：调用模型接口。
- 为什么需要：隔离模型调用逻辑，后续便于替换 OpenAI、DeepSeek、通义等模型。

5. `answer_project_question(repo_path: str, question: str) -> dict`

- 输入：项目路径、用户问题。
- 输出：包含回答和参考文件路径的结果字典。
- 作用：串联扫描、选文件、读文件、构造 prompt、调用 LLM。
- 为什么需要：作为 V1.5 的主流程函数，方便 CLI 或后续 Agent 节点复用。

---

### 8. Prompt 设计要求

Prompt 至少应包含：

```text
你是一个代码仓库分析助手。

请只基于我提供的项目结构和文件内容回答。
如果上下文不足，请明确说明“不确定”或“当前提供的信息不足”。
回答中尽量引用相关文件路径。

项目结构：
{tree}

文件类型统计：
{file_types}

入口文件候选：
{entry_candidates}

关键文件内容：
{file_contents}

用户问题：
{question}
```

关键约束：

- 不要让模型凭空猜测。
- 不要声称看过未提供的文件。
- 尽量输出文件路径依据。
- 如果上下文不足，要说明下一步应该读取哪个文件。

---

### 9. 错误处理与边界

错误处理：

- 路径不存在：返回明确错误。
- 路径不是目录：返回明确错误。
- 没有找到关键文件：仍然可以只基于目录树回答。
- LLM API Key 缺失：提示用户配置环境变量。
- LLM 调用失败：返回错误原因。

边界约束：

- 每个文件默认最多读取 `8,000` 字符。
- 所有上下文总长度建议不超过 `30,000` 字符。
- 默认最多选择 `8` 个关键文件。
- 超限时优先保留 README、入口文件、配置文件。

---

### 10. 验收标准（DoD）

使用测试仓库：

```text
E:\projects\test_project_v1
```

通过标准：

- 可以通过 CLI 输入项目路径和问题。
- 能复用 V1 的项目结构扫描结果。
- 能读取 README、入口文件、配置文件等关键文件。
- 能调用 LLM 返回自然语言回答。
- 回答中包含使用到的文件路径。
- 当上下文不足时，能明确说明还需要读取哪些文件。
- 不依赖 embedding、FAISS、Chroma 或其他向量数据库。

---

### 11. 在 Agent 系统中的角色

V1.5 模块归类：

- Prompt 模板
- LLM 调用模块
- 文件工具
- 工程辅助模块

它的定位：

- 不是完整 Agent。
- 不是 RAG 系统。
- 是“代码仓库上下文 + LLM 回答”的最小验证版本。
- 为后续 RAG、Tool Calling、LangGraph 节点提供实验基础。

---

### 12. 对应经典项目设计思想

- LangChain：类似 `PromptTemplate + LLMChain` 的早期形态，先把上下文拼进 prompt，再调用模型生成回答。
- LangGraph：后续可以把 `answer_project_question` 拆成节点，例如 `scan_repo -> build_context -> call_llm -> answer`。
- LlamaIndex：暂时不使用索引，但已经有了 `documents/context` 的思想，后续可以把关键文件升级为 `Document` 和 `Node`。
- OpenAI Agents SDK：类似最基础的模型调用流程，后续可升级为带 tools、tracing、guardrails 的 Agent。

---

### 13. 后续如何升级

V1.5 -> V2：

- 将关键文件读取升级为全项目代码加载。
- 增加代码切分。
- 增加 embedding。
- 增加向量索引。
- 将“手动选文件”升级为“根据问题自动检索相关代码片段”。

V1.5 -> V3：

- 将读取文件、搜索代码、总结文件封装成 tools。
- 让 LLM 根据问题主动选择工具。

V1.5 -> V4：

- 将流程拆成 LangGraph 节点。
- 引入 `AgentState` 保存扫描结果、用户问题、上下文文件、模型回答和执行步数。
