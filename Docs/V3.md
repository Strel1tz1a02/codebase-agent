# V3: Tool Calling Agent 具体规划

## 目标

V3 的目标是把当前项目从“被动问答工具”升级为“可以由 LLM 决策并调用本地工具的最小 Agent”。

用户提出问题后，系统不再只把上下文交给 LLM 直接回答，而是允许 LLM 先选择工具，例如查看仓库概况、读取文件、搜索代码、检索相关代码片段，再基于工具结果生成最终回答。

当前已经跑通的最小链路：

```text
用户问题
-> build_prompt
-> LLM 返回 JSON 决策
-> parse_llm
-> run_agent_loop
-> execute_tool
-> 工具结果写入 history
-> LLM 基于 history 继续回答
```

已验证工具：

```text
repo_summary
```

它可以返回仓库文件数、主要目录、文件类型统计和入口候选文件，用于证明 LLM 到工具调用的完整链路已经可运行。

---

## V3 完成标准

V3 不追求复杂 Agent 框架，重点是把 Tool Calling 的核心机制做清楚。

完成标准：

1. LLM 能通过 JSON 决策调用工具。
2. 本地工具能接收参数并返回结构化结果。
3. 工具执行结果会写入 Agent history。
4. LLM 能基于 history 中的工具结果继续生成答案。
5. 工具失败时有明确错误信息，不会让程序崩溃。
6. CLI 可以通过 `--ask-mode agent` 触发 Agent 流程。
7. 每个新增工具都有单元测试。

## 总体架构

V3 保持当前简单分层：

```text
src/main.py
  负责 CLI 参数解析，选择 basic / rag / agent 模式。

src/agent/adapter.py
  负责构造 LLM prompt，调用 LLM，并把 LLM 输出解析成 decision payload。

src/agent/controller.py
  负责 Agent 循环，维护 context/history，控制最大步数。

src/agent/tools.py
  负责工具注册、按名称分发工具、包装工具成功或失败结果。

src/agent/schemas.py
  负责 AgentContext、AgentDecision、ToolResult 和 decision 校验。

src/tools/file_tools.py
  复用已有 V1 仓库扫描能力。

src/rag/
  复用已有 V2 代码检索能力。
```

V3 暂时不引入 LangGraph。LangGraph 放到 V4 做，因为当前阶段更重要的是先把工具调用协议、工具执行、history 回传跑稳。

## 第 1 步：固定 Agent 决策协议（已完成）

### 要做什么

明确 LLM 只能返回两种 JSON：

```json
{"decision":"tool","tool_name":"repo_summary","arguments":{}}
```

或：

```json
{"decision":"answer","answer":"项目入口位于 src/main.py"}
```

### 涉及文件

```text
src/agent/adapter.py
src/agent/schemas.py
tests/test_agent_adapter.py
```

### 具体任务

1. 在 `build_prompt()` 中说明可用工具列表和返回格式。
2. 在 `parse_llm()` 中支持纯 JSON 和 ```json fenced block。
3. 在 `validate_decision_payload()` 中校验：
   - `decision` 只能是 `tool` 或 `answer`
   - `decision=tool` 时必须有 `tool_name`
   - `decision=tool` 时必须有 `arguments`
   - `decision=answer` 时必须有 `answer`
4. 给解析失败、字段缺失、格式错误分别写测试。

### 验证命令

```powershell
python -m unittest tests.test_agent_adapter
```

## 第 2 步：实现最小 Agent 循环（已完成）

### 要做什么

让 Agent 能在多轮中执行：

```text
拿 context 给 LLM
-> LLM 决定调用工具或直接回答
-> 如果调用工具，执行工具并写入 history
-> 下一轮把 history 再给 LLM
-> 直到 LLM 返回 answer 或达到 max_steps
```

### 涉及文件

```text
src/agent/controller.py
src/agent/schemas.py
tests/test_agent_controller.py
```

### 具体任务

1. 创建 `AgentContext`，保存：
   - `question`
   - `repo_path`
   - `allowed_tools`
   - `history`
2. 实现 `run_agent_loop()`。
3. 支持 `max_steps`，防止无限循环。
4. 工具调用前记录 LLM decision。
5. 工具调用后记录 tool result。
6. LLM 直接回答时返回：

```python
{
    "status": "completed",
    "answer": "...",
    "history": [...]
}
```

7. 超过最大步数时返回：

```python
{
    "status": "stopped",
    "answer": "",
    "reason": "max_steps reached",
    "history": [...]
}
```

### 验证命令

```powershell
python -m unittest tests.test_agent_controller
```

## 第 3 步：实现工具执行器（已完成）

### 要做什么

实现工具注册表和统一执行入口。

工具执行器只负责三件事：

1. 根据工具名找到函数。
2. 把 arguments 传给工具。
3. 把成功或失败包装成 `ToolResult`。

### 涉及文件

```text
src/agent/tools.py
src/agent/schemas.py
tests/test_agent_executor.py
```

### 具体任务

1. 定义 `TOOL_REGISTRY`。
2. 实现 `execute_tool(tool_name, arguments)`。
3. 未知工具返回：

```python
ToolResult(
    ok=False,
    tool_name=tool_name,
    output={},
    error=f"unknown tool: {tool_name}",
)
```

4. 工具内部异常要被捕获并转成失败结果。
5. 工具成功时返回：

```python
ToolResult(
    ok=True,
    tool_name=tool_name,
    output=output,
    error="",
)
```

## 第 6 步：实现 search_code 工具（已完成）

### 要做什么

让 Agent 可以按关键词搜索代码文件，定位相关文件和行号。

工具名：

```text
retrieve_code
```

输入：

```json
{"keyword":"run_agent_loop", "scope":"src"}
```

输出：

```python
{
    "keyword": "run_agent_loop",
    "scope": "src",
    "matches": [
        {
            "path": "src/agent/controller.py",
            "line": 8,
            "text": "def run_agent_loop("
        }
    ]
}
```

### 涉及文件

```text
src/agent/tools.py
src/agent/controller.py
src/agent/adapter.py
tests/test_agent_executor.py
tests/test_agent_controller.py
```

### 具体任务

1. 新增 `search_code(arguments)`。
2. 参数为 `keyword`。
3. 支持 `scope` 参数，默认 `src`。
4. `scope=src` 只搜索 `src/`。
5. `scope=tests` 只搜索 `tests/`。
6. `scope=docs` 只搜索 `Docs/` 和 `README.md`。
7. `scope=all` 搜索全仓库可搜索文本文件。
8. 只搜索文本代码文件，先覆盖 `.py`、`.md`、`.txt`、`.json`。
9. 忽略 `.git`、`__pycache__`、`.pytest_cache` 等目录。
10. 每次最多返回 20 条结果。
11. 每条结果包含：
   - 相对路径
   - 行号
   - 当前行文本
12. 在 prompt 中告诉 LLM：
   - 不知道代码位置时先调用 `search_code`
   - 默认使用 `scope=src`
   - 用户明确问测试时使用 `scope=tests`
   - 用户明确问文档时使用 `scope=docs`
   - 用户明确要求全仓库时使用 `scope=all`
13. 测试默认搜索 `run_agent_loop` 能命中 `src/agent/controller.py`。
14. 测试默认搜索不返回 `Docs/`、`README.md`、`tests/`。
15. 测试搜索不存在关键词时返回空 matches。

## 推荐实现顺序

按这个顺序推进：

```text
1. repo_summary
2. read_file
3. search_code
4. retrieve_code
5. prompt 工具选择规则
6. CLI 输出优化
7. 全量测试和 README 更新
```

原因：

1. `repo_summary` 最简单，已经证明链路可用。
2. `read_file` 是所有代码理解工具的基础。
3. `search_code` 适合精确定位函数、类、关键词。
4. `retrieve_code` 复用 V2，适合自然语言问题。
5. 工具变多后再优化 prompt，更容易根据真实问题调整。
6. CLI 输出最后整理，避免前期频繁改展示格式。
