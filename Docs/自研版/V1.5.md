## V1.5：LLM 项目问答实验版（不使用 RAG）

### 1. 当前模块目标

V1.5 的目标是在不引入 RAG、向量库和复杂 Agent 编排的前提下，先打通“基于有限项目上下文的问答”主流程：

1. 复用 V1 项目结构扫描结果。
2. 选择少量关键文件并读取内容。
3. 组装 prompt。
4. 调用 LLM 返回自然语言回答。
5. 输出回答及使用到的上下文文件路径。

V1.5 是 V1 与 V2 之间的过渡版本，用于验证最小可用问答能力。

---

### 2. 范围定义

`In Scope`（V1.5 要做）

1. 支持 CLI 输入项目路径和问题。
2. 复用 V1 的扫描信息（目录树、文件类型、入口候选等）。
3. 自动读取关键文件（README、配置、入口文件等）。
4. 构建项目问答 prompt。
5. 调用 LLM 并返回回答。
6. 返回并展示 `used_files`。

`Out of Scope`（V1.5 不做）

1. 不做代码切片。
2. 不做 embedding。
3. 不做向量数据库。
4. 不做语义检索。
5. 不做多轮 Agent 状态编排。
6. 不做 LangGraph 节点编排。

---

### 3. 当前实现结构

当前核心文件：

1. `src/tools/file_tools.py`
2. `src/prompts/templates.py`
3. `src/llm/client.py`
4. `src/qa.py`
5. `src/main.py`

职责关系：

1. `file_tools.py`：扫描后关键文件选择与读取。
2. `templates.py`：构建最终问答 prompt。
3. `client.py`：统一 LLM 调用入口与提供商路由。
4. `qa.py`：串联完整问答流程。
5. `main.py`：CLI 参数解析与输出。

---

### 4. 主流程（当前代码）

`answer_project_question(scan_result, question)` 的执行顺序：

1. `select_context_files(scan_result, repo_path)`
2. `read_context_files(selected_files)`
3. `build_project_qa_prompt(repo_summary, question, file_contents)`
4. `ask_llm(prompt)`
5. 返回：
   - `answer: str`
   - `used_files: list[str]`
   - `prompt: str`

---

### 5. 关键函数定义（当前）

1. `select_context_files(scan_result, repo_path, max_files=8) -> list[str]`
   - 规则优先选：`README.md`、`pyproject.toml`、`requirements.txt`、`package.json`、入口文件与 `entry_candidates`。
   - 仅返回真实文件。
   - 去重并保持稳定顺序。

2. `read_context_files(file_paths, max_chars_per_file=8000) -> dict[str, str]`
   - UTF-8 读取。
   - 单文件字符数截断。
   - 不存在文件跳过。
   - 编码/IO 异常返回错误文本，不抛出到上层。

3. `build_project_qa_prompt(repo_summary, question, file_contents=None) -> str`
   - 包含：项目结构、文件统计、入口候选、关键文件内容、用户问题。
   - 未提供 `file_contents` 时使用占位提示。

4. `ask_llm(prompt) -> str`
   - 使用 OpenAI Python SDK 调用兼容接口。
   - 由 `configure_llm(...)` 注入运行时配置。
   - 支持已注册 provider 路由与模型命中校验。

5. `answer_project_question(scan_result, question) -> dict[str, object]`
   - 返回 `answer/used_files/prompt`，供 CLI 与后续流程复用。

---

### 6. LLM 配置策略（当前）

当前实现为“命令行参数驱动”，不依赖环境变量：

1. `--provider`
2. `--model`
3. `--api-key`
4. `--base-url`

`src/main.py` 在 `--ask` 流程中调用 `configure_llm(...)` 注入配置。

---

### 7. Provider 与模型注册（当前）

`src/llm/client.py` 中维护：

1. `REGISTERED_PROVIDERS`
2. `PROVIDER_MODEL_REGISTRY`

行为规则：

1. provider 未注册：直接报错。
2. model 为空：直接报错。
3. model 未命中当前 provider 注册表：直接报错。
4. API 异常：返回可读错误，避免 CLI 崩溃。

---

### 8. 命令行使用方式（当前）

仅扫描：

```bash
python src/main.py --repo E:\projects\codebase-agent
```

问答（示例：阿里云兼容模式）：

```bash
python src/main.py --repo E:\projects\codebase-agent --ask "这个项目入口在哪里？" --provider aliyun --model qwen-plus --api-key <你的key> --base-url https://dashscope.aliyuncs.com/compatible-mode/v1
```

输出包含三段：

1. `## Prompt`
2. `## 回答`
3. `## 使用的上下文文件`

---

### 9. 验收口径（按当前实现）

通过标准：

1. 能通过 CLI 传入项目路径和问题执行问答。
2. 能复用 V1 扫描结果。
3. 能自动选择并读取关键文件。
4. 能构建包含关键文件内容的 prompt。
5. `ask_llm` 为真实 API 调用入口（非 TODO 占位）。
6. `answer_project_question` 返回结构包含 `answer/used_files/prompt`。
7. 配置错误、模型未命中、网络异常均不会导致主流程崩溃。
8. 不引入 RAG、embedding、向量库。
