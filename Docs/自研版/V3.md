# V3: Tool Calling Agent 收尾文档

## 目标

V3 的目标是把项目从“被动问答工具”升级为“可以由 LLM 决策并调用本地工具的最小 Agent”。

用户提出问题后，系统允许 LLM 先选择工具，例如查看仓库概况、读取文件、搜索代码，再基于工具结果生成最终回答。

当前 V3 已经跑通的链路：

```text
用户问题
-> build_prompt
-> LLM 返回 JSON 决策
-> parse_llm
-> run_agent_loop
-> execute_tool
-> 工具结果写入 history
-> LLM 基于 history 继续回答
```

## 完成结论

V3 核心功能已完成，可以作为基础 Tool Calling Agent 版本收尾。

已满足的完成标准：

1. LLM 能通过 JSON 决策调用工具。
2. 本地工具能接收参数并返回结构化结果。
3. 工具执行结果会写入 Agent history。
4. LLM 能基于 history 中的工具结果继续生成答案。
5. 工具失败时有明确错误信息，不会让程序崩溃。
6. CLI 可以通过 `--ask-mode agent` 触发 Agent 流程。
7. 新增工具均有单元测试。

## 当前架构

```text
src/main.py
  负责 CLI 参数解析，选择 basic / rag / agent 模式。

src/agent/adapter.py
  负责构造 LLM prompt，调用 LLM，并把 LLM 输出解析成 decision payload。

src/agent/controller.py
  负责 Agent 循环，维护 context/history，控制最大步数，并统一给工具参数注入 repo_path。

src/agent/tools.py
  负责工具注册、按名称分发工具、真实工具实现、工具成功/失败结果包装。

src/agent/schemas.py
  负责 AgentContext、AgentDecision、ToolResult 和 decision 校验。

src/tools/file_tools.py
  复用 V1 仓库扫描与忽略规则。

src/rag/
  保留 V2 RAG 检索能力，后续可接入 Agent。
```

## 已实现能力

### Agent 决策协议

LLM 只能返回两类 JSON。

调用工具：

```json
{"decision":"tool","tool_name":"repo_summary","arguments":{}}
```

直接回答：

```json
{"decision":"answer","answer":"项目入口位于 src/main.py"}
```

### Agent 循环

`run_agent_loop()` 支持：

1. 构造 `AgentContext`。
2. 把 `question`、`repo_path`、`allowed_tools`、`history` 交给 LLM。
3. 执行 LLM 选择的工具。
4. 把工具结果写入 `history`。
5. 继续下一轮，直到 LLM 返回 answer 或达到 `max_steps`。

### 工具执行层

`execute_tool()` 支持：

1. 根据工具名从 `TOOL_REGISTRY` 找工具。
2. 把 arguments 传给工具。
3. 捕获工具异常。
4. 返回统一 `ToolResult`。

未知工具会返回：

```text
unknown tool: <tool_name>
```

### 默认仓库路径注入

Controller 会统一给所有工具调用注入 `repo_path`：

```python
arguments = dict(decision["arguments"])
if "repo_path" not in arguments:
    arguments["repo_path"] = repo_path
```

工具层仍然保留 `repo_path` 校验，因为工具可能被测试、脚本或未来其它入口直接调用。

## 已实现工具

### repo_summary

用途：查看仓库概况和入口候选。

输入：

```json
{"repo_path":"E:\\projects\\codebase-agent"}
```

输出包含：

```text
repo_path
file_count
file_types
key_dirs
entry_candidates
```

该工具复用 V1 的 `run_v1_scan()`。

### read_file

用途：读取仓库内指定文件内容。

输入：

```json
{"path":"src/main.py"}
```

输出包含：

```text
path
content
truncated
```

安全规则：

1. 只能读取仓库内文件。
2. 路径不能逃出仓库，例如 `..\secret.txt` 会失败。
3. 默认最多读取 8000 字符。

### search_code

用途：按关键词搜索文本代码文件，返回相对路径、行号和当前行文本。

输入：

```json
{"keyword":"run_agent_loop", "scope":"src"}
```

输出包含：

```text
keyword
scope
matches
```

每条 match 包含：

```text
path
line
text
```

搜索范围：

```text
src    默认，只搜索 src/
tests  只搜索 tests/
docs   只搜索 Docs/ 和 README.md
all    搜索全仓库可搜索文本文件
```

默认使用 `scope=src`，避免普通代码问题被 `Docs/README/tests` 结果干扰，也能减少第二次 LLM 调用的 token。

该工具复用 V1 的 `scan_files()`，因此会继承 `.git`、`__pycache__`、缓存目录等忽略规则。


## 手动验收命令

查看仓库概况并定位入口：

```powershell
python src/main.py --repo E:\projects\codebase-agent --ask-mode agent --ask "请先调用工具查看这个仓库的基本情况，然后回答项目入口在哪里" --max-steps 3
```

读取入口文件：

```powershell
python src/main.py --repo E:\projects\codebase-agent --ask-mode agent --ask "请读取 src/main.py，然后告诉我 main 函数做了什么" --max-steps 4
```

搜索函数位置：

```powershell
python src/main.py --repo E:\projects\codebase-agent --ask-mode agent --ask "请搜索 run_agent_loop，并告诉我它在哪个文件里" --max-steps 4
```

预期 `History` 中可以看到：

```text
'decision': 'tool'
'tool_name': 'search_code'
'ok': True
```

## 后续增强项

下面内容不是 V3 收尾的阻塞项，可以放到后续版本或增强任务中。

1. `retrieve_code`：把 V2 RAG 检索接入 Agent。
2. Prompt 优化：进一步细化工具选择规则。
3. CLI 输出优化：让 history 展示更紧凑。
4. V4：用 LangGraph 重构 Agent 流程。

## 建议版本边界

V3 到此可以定义为：

```text
基础 Tool Calling Agent
```

它已经具备：

1. LLM 决策。
2. 工具调用。
3. 工具结果回传 history。
4. 真实仓库工具。
5. 测试覆盖。
6. CLI 验证入口。

下一阶段建议先保留 V3 稳定状态，再决定是增强 `retrieve_code`，还是进入 V4 LangGraph。
