# V6: Agent Runtime 与 Session Memory

## 当前模块目标

V6 的目标是把一次性 Agent runner 升级为轻量级 Runtime。Runtime 负责创建会话、保存多轮消息、调用 Agent runner、记录运行摘要，并把 stop reason 返回给调用方。

本阶段不新增 API，不改 CLI 入口，也不实现数据库持久化。当前实现使用进程内存保存 session，方便先学习 Runtime 的状态边界。

## 推荐文件位置

本阶段新增 Runtime 层：

1. `src/runtime/session.py`
   - 定义 `Message`、`Trace`、`Session`。
2. `src/runtime/memory.py`
   - 定义 `SessionMemory`，负责创建和查找 session。
3. `src/runtime/runtime.py`
   - 定义 `AgentRuntime`，负责串联 session 与 agent runner。
4. `src/runtime/graph_runner.py`
   - 定义 `build_graph_agent_runner()`，把 Runtime payload 适配到 `run_agent_graph()`。
5. `tests/test_runtime_*.py`
   - 覆盖 session、memory、runtime 和 graph runner。

## 代码实现

V6 当前核心链路：

```text
SessionMemory.create_session(repo_path)
-> AgentRuntime.ask(session_id, question)
-> Session.append_message("user", question)
-> agent_runner(payload)
-> Session.append_message("assistant", answer)
-> Session.append_trace({"status": ..., "answer": ..., "reason": ...})
```

Runtime 传给 agent runner 的 payload：

```json
{
  "question": "入口在哪？",
  "repo_path": "E:\\projects\\codebase-agent",
  "messages": [
    {"role": "user", "content": "入口在哪？"}
  ]
}
```

## 函数解释

### `Session.append_message(role, content)`

输入：

1. `role`：消息角色，目前为 `user` 或 `assistant`。
2. `content`：消息正文。

输出：

1. `Message`：追加后的消息对象。

作用：

把一条对话消息追加到当前 session。

为什么需要：

多轮会话的历史属于 session 自己的内部状态，因此追加消息的规则放在 `Session` 层，而不是放在 memory 或 runtime 层。

### `Message.to_dict()` / `Trace.to_dict()` / `Session.message_dicts`

输入：

无。

输出：

普通字典形式的消息快照：

```python
[{"role": "user", "content": "..."}]
```

Runtime/API 调用集合快照时使用：
```python
session.message_dicts
session.trace_dicts
```

作用：

把内部 `Message` 对象转换为 agent runner 更通用的 payload。

为什么需要：

runner 不应该依赖 `Message` dataclass。用普通 dict 可以降低耦合，并避免外部直接修改 session 内部对象。

### `SessionMemory.create_session(repo_path)`

输入：

1. `repo_path`：当前 session 绑定的仓库路径。

输出：

1. `Session`：新创建的 session。

作用：

创建一个新的会话，并用 `session_id` 存入内存。

为什么需要：

Runtime 需要通过 `session_id` 找回同一次对话的仓库路径、消息历史和 trace。

### `AgentRuntime.ask(session_id, question)`

输入：

1. `session_id`：已有 session ID。
2. `question`：用户本轮问题。

输出：

字典结果，包含：

1. `session_id`
2. `status`
3. `question`
4. `answer`
5. `reason`
6. `message_count`

作用：

把用户问题写入 session，调用 agent runner，保存 assistant 回答和运行摘要。

为什么需要：

V3/V4 的 runner 只处理一次请求。Runtime 是更上层的会话运行边界，用来支撑多轮追问和未来 API。

### `build_graph_agent_runner(...)`

输入：

1. `llm_decision_func`：LLM 决策函数。
2. `max_steps`：最大工具调用步数。
3. `run_graph_func`：默认是 `run_agent_graph()`。

输出：

1. `agent_runner(payload)`：可注入 `AgentRuntime` 的 runner。

作用：

把 Runtime payload 适配成 `run_agent_graph()` 的参数。

为什么需要：

`AgentRuntime` 不直接依赖 LangGraph 函数签名。以后如果接入 LangGraph checkpointer、异步 runner 或 eval runner，只需要替换适配层。

## 在 Agent 系统中的角色

V6 属于以下能力：

1. Agent Runtime。
2. Session Memory。
3. Agent State 管理。
4. 日志 / 配置 / 工程辅助模块中的运行摘要能力。

## 对应经典项目设计思想

1. LangGraph
   - 类似 thread-level memory / checkpointer 的上层平台抽象，但当前实现只做教学版内存 session。
   - `graph_runner.py` 类似 graph invocation adapter，把外部 runtime payload 转成 graph 调用参数。
2. LangChain
   - 类似 memory/message history：保存 user/assistant 消息，并在下一轮调用时传给模型。
3. OpenAI Agents SDK
   - 类似 run/session 层：Runtime 负责接收用户输入、调用 agent、保存输出。
4. LlamaIndex
   - 类似 chat engine 的 conversation history，历史消息作为下一轮 query 的上下文。

## 当前取舍

### 为什么不直接使用 LangGraph checkpointer

LangGraph checkpointer 更适合保存 graph 内部 state。V6 当前要先学习平台层 Runtime 的边界：session、messages、trace、runner adapter。后续可以把 `SessionMemory` 替换或扩展为 LangGraph checkpointer、SQLite 或其他持久化方案。

### 为什么 V6 暂不实现 retry policy

PlanA 中 V6 提到 retry policy，但当前阶段先不做。主流实践更倾向于在具体失败点 retry，例如 LLM client、graph node 或 tool executor。当前项目的细粒度工具协议将在 V8 实现，因此 retry 暂时放到 V8 或后续 Runtime 增强中，避免现在做粗粒度整轮重试导致重复工具调用。

## 后续如何升级

1. V7：把 `AgentRuntime` 暴露为 FastAPI session 接口。
2. V8：在 MCP 风格工具协议中加入 tool-level retry、schema 和 permission。
3. V9：基于 session trace 和 eval cases 统计 Agent 成功率、引用覆盖率和失败原因。
4. 后续增强：将 `SessionMemory` 从内存 dict 替换为持久化 store 或 LangGraph checkpointer。
