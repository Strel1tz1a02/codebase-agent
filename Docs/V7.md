# V7: FastAPI 服务

## 当前模块目标

V7 的目标是把 V6 已经完成的 `AgentRuntime` 暴露为一个 FastAPI 后端服务。

V6 解决的是 Python 进程内部的 runtime 问题：

```text
AgentRuntime
-> SessionMemory
-> Session
-> graph_runner
```

V7 要解决的是 HTTP 服务边界问题：

```text
HTTP Request
-> FastAPI endpoint
-> AgentRuntime
-> HTTP Response
```

本阶段不是重写 Agent，也不是改 RAG 效果，而是给现有 runtime 增加一个稳定的 API 外壳。

## 推荐文件位置

V7 推荐新增：

```text
src/api/__init__.py
src/api/schemas.py
src/api/app.py
tests/test_api_app.py
```

文件职责：

1. `src/api/schemas.py`
   - 定义请求和响应 schema。
   - 让 API 输入输出结构明确。
2. `src/api/app.py`
   - 创建 FastAPI app。
   - 定义 API 路由。
   - 持有一个进程内 `AgentRuntime`。
3. `tests/test_api_app.py`
   - 使用 FastAPI TestClient 测试 HTTP 行为。
   - 第一版测试用 fake runner，避免真实 LLM 影响测试稳定性。

暂时不新增复杂 service 层。V6 的 `AgentRuntime` 已经是服务边界，API 层只负责 HTTP 适配。

## API 设计

### `GET /health`

作用：

检查服务是否可用。

响应示例：

```json
{
  "status": "ok"
}
```

### `GET /tools`

作用：

列出当前可用 Agent 工具。

第一版可以直接返回 `TOOL_REGISTRY.keys()`。

响应示例：

```json
{
  "tools": [
    "repo_summary",
    "read_file",
    "search_code",
    "retrieve_code"
  ]
}
```

### `POST /sessions`

作用：

创建一个绑定到指定仓库路径的 session。

请求示例：

```json
{
  "repo_path": "E:\\projects\\codebase-agent"
}
```

响应示例：

```json
{
  "session_id": "abc123",
  "repo_path": "E:\\projects\\codebase-agent",
  "status": "running",
  "message_count": 0
}
```

### `GET /sessions/{session_id}`

作用：

查看指定 session 的状态、消息历史和 trace 摘要。

响应示例：

```json
{
  "session_id": "abc123",
  "repo_path": "E:\\projects\\codebase-agent",
  "status": "completed",
  "messages": [
    {"role": "user", "content": "入口在哪？"},
    {"role": "assistant", "content": "入口在 src/main.py。"}
  ],
  "trace": [
    {
      "status": "completed",
      "answer": "入口在 src/main.py。",
      "reason": ""
    }
  ]
}
```

如果 session 不存在，返回 404。

### `POST /sessions/{session_id}/ask`

作用：

在指定 session 中提问。

请求示例：

```json
{
  "question": "入口在哪？"
}
```

响应示例：

```json
{
  "session_id": "abc123",
  "status": "completed",
  "question": "入口在哪？",
  "answer": "入口在 src/main.py。",
  "message_count": 2
}
```

如果 Agent 停止或失败，响应里包含 `reason`。

## 分步实现计划

### Step 1: 增加依赖

修改：

```text
requirements.txt
```

新增：

```text
fastapi
uvicorn
httpx
```

说明：

1. `fastapi` 用于创建 API。
2. `uvicorn` 用于本地运行服务。
3. `httpx` 是 FastAPI `TestClient` 的常见测试依赖。

### Step 2: 定义 API schema

新增：

```text
src/api/schemas.py
```

建议 schema：

```python
class CreateSessionRequest(BaseModel):
    repo_path: str

class CreateSessionResponse(BaseModel):
    session_id: str
    repo_path: str
    status: str
    message_count: int

class AskRequest(BaseModel):
    question: str

class AskResponse(BaseModel):
    session_id: str
    status: str
    question: str
    answer: str = ""
    reason: str = ""
    message_count: int

class SessionResponse(BaseModel):
    session_id: str
    repo_path: str
    status: str
    messages: list[dict[str, str]]
    trace: list[dict[str, object]]

class HealthResponse(BaseModel):
    status: str

class ToolListResponse(BaseModel):
    tools: list[str]
```

### Step 3: 创建 FastAPI app

新增：

```text
src/api/app.py
```

第一版 app 提供一个构造函数：

```python
def create_app(runtime: AgentRuntime | None = None) -> FastAPI:
    ...
```

为什么需要 `create_app()`：

测试可以传入 fake runtime 或 fake runner，避免真实 LLM 调用。

模块级默认 app：

```python
app = create_app()
```

这样 uvicorn 可以直接运行：

```powershell
uvicorn src.api.app:app --reload
```

### Step 4: 先实现非 LLM 接口

第一小步只实现：

```text
GET /health
GET /tools
POST /sessions
GET /sessions/{session_id}
```

暂时不实现 `/ask`。

原因：

这样可以先学习 FastAPI 如何对接 Runtime 的 session 管理，不引入 LLM 和 LangGraph 变量。

### Step 5: 再实现 ask 接口

第二小步实现：

```text
POST /sessions/{session_id}/ask
```

测试用 fake runner：

```python
def fake_runner(payload):
    return {
        "status": "completed",
        "answer": "入口在 src/main.py。",
        "history": [],
    }
```

这样可以稳定验证：

1. API 能找到 session。
2. API 能调用 runtime.ask。
3. user/assistant messages 被保存。
4. answer/status/reason 能返回给 HTTP 调用方。

### Step 6: 接默认 graph runner

当测试稳定后，默认 app 可以构造：

```python
AgentRuntime(
    agent_runner=build_graph_agent_runner(
        llm_decision_func=next_decision,
        max_steps=3,
    )
)
```

注意：

真实 LLM 需要读取配置和 API key。V7 第一版可以先保持测试使用 fake runner，默认运行时沿用现有 LLM 配置方式，后续再优化配置注入。

## 测试计划

新增：

```text
tests/test_api_app.py
```

测试场景：

1. `GET /health` 返回 `{"status": "ok"}`。
2. `GET /tools` 返回已有工具名。
3. `POST /sessions` 创建 session。
4. `GET /sessions/{session_id}` 返回 session 状态。
5. 查询不存在 session 返回 404。
6. `POST /sessions/{session_id}/ask` 返回 answer。
7. ask 后 `GET /sessions/{session_id}` 能看到 user/assistant messages。
8. ask 返回 stopped/failed 时包含 reason。

## 当前不做

V7 第一版不做：

1. 用户认证。
2. 数据库持久化。
3. 多 worker 共享 session。
4. 异步后台任务队列。
5. WebSocket 或流式输出。
6. 复杂配置中心。
7. 浏览器 UI。

这些可以作为后续工程化增强，不应干扰当前 API 服务边界学习。

## 在 Agent 系统中的角色

V7 属于：

1. Python 后端服务。
2. API 服务层。
3. Agent Runtime 的 HTTP 适配层。
4. 后续前端、评测和部署的基础入口。

## 对应经典项目设计思想

1. FastAPI
   - 路由函数负责 HTTP 输入输出。
   - Pydantic schema 负责请求和响应结构。
2. LangGraph
   - API 层不直接操作 graph state，而是通过 Runtime 调用 graph runner。
3. OpenAI Agents SDK
   - 类似把 agent run 封装成服务端接口，外部只关心 session 和 ask。
4. LangChain / LlamaIndex
   - 类似把 chain/chat engine 暴露成 REST API。
